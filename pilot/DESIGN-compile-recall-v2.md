# 컴파일 recall v2 — 지침 개선 재측정 사전등록 (2026-07-12, 이슈 #14)

v1(RESULTS-compile-recall.md)이 미스 상위 요인으로 지목한 ① 구조(표·코드블록 안의
룰 미승격, 45건) ② 번들 룰 1:1 페널티에 대해 `commands/compile.md` 지침을 수정하고,
**같은 코퍼스·같은 gold**로 재컴파일→재매칭해 변화를 잰다. 실행 전에 처치·지표·
비교 규칙을 고정한다.

## 1. 처치 (지침 수정, 이 문서와 같은 커밋으로 고정)

`commands/compile.md` §2에 두 가지를 추가한다:

- **구조 승격**: 표·코드블록·리스트 안의 내용도 본문 문장과 동등한 룰 후보.
  단 사용법 카탈로그(무엇을 실행할 수 있는지 나열)와 제약(의무·금지·순서)을
  구분하고, 제약이 붙은 것만 룰로 승격한다.
- **번들 분리(좁게)**: 한 문장이 **트리거가 서로 다른** 요구를 묶고 있으면 분리한다.
  같은 트리거·같은 strength·같은 원문의 나열(예: langgraph의 format/lint/test)은
  분리하지 **않는다** — 같은 문서가 여러 번 배달되는 런타임 비용만 생긴다.
  따라서 v1에서 1:1 페널티로 집계된 미스(langgraph 2건 등)는 v2에서도 미스로
  남을 것으로 예상하며, 이는 실패가 아니라 의도된 설계다.

## 2. 실행 프로토콜

- 코퍼스: v1과 동일 4건 (manifest SHA 검증, 로컬 사본 재사용).
- 컴파일: DESIGN-compile-bench §3과 동일 — 문서당 서브에이전트 1개, **sonnet 고정**,
  수정된 `commands/compile.md` §2~§4를 프롬프트에 포함, 출력 스키마 동일(§3.1).
  오류·스키마 불일치 시 1회 재시도, 실패면 해당 문서 제외·사유 기록.
- 출력 저장: `pilot/compile-recall/outputs-v2/<id>.json` (v1 outputs는 불변).
- 매칭: v1 A1 프로토콜 그대로 — Codex 새 세션/문서, 격리 디렉터리, 네트워크 차단.
  기준 문서도 v1과 동일(MATCH-CRITERIA).
- 집계: `grade_compile_recall.py`에 outputs/matches 디렉터리 인자를 추가하되
  기본값은 v1 경로로 유지(v1 재현성 보존).

## 3. 비교 규칙 (v1 ↔ v2)

- **gold는 93da52e 기준 181룰로 고정하고 v2에서는 갱신하지 않는다** — 분모가
  움직이면 M1이 비교 불능이 된다. v2 매칭에서 gold-miss가 발견되면
  unmatched_extracted에 기록만 하고(M3), 정답에 추가하지 않는다.
- M1 비교는 마이크로 recall (v1 = 62/181 = 34%) 및 문서별.
- **게이트 (v1에 없던 것)**: v2에서 over-extraction이 0에서 벗어나면 지침 수정은
  개선이 아니다 — recall이 올라도 채택하지 않고 지침을 되돌린다.
- 형식 품질 재검: v2 룰 전수에 대해 정규식 컴파일, tool 어휘, content 룰 `path:`
  동반, compile-bench M3 고정 무해 커맨드 10종 스모크를 재적용한다. 여기서
  후퇴(v1 대비 신규 위반)가 나오면 게이트와 동일하게 취급한다.

## 4. 해석 한계

- 판정자(Codex)는 v1과 같은 프로토콜이지만 다른 세션이다 — 판정 드리프트가
  M1 변화에 섞일 수 있다. v2 매칭 파일의 근거(rationale/note)를 전부 남겨
  사람이 검수한다.
- 문서당 1런·모델 1종 — v1과 동일하게 분산 추정 불가, 파일럿.
- gold 고정의 대가: v2가 새로 찾은 진짜 룰은 M1에 반영되지 않는다(M3에만 기록).
  M1은 "고정된 181룰 기준"으로만 해석한다.
