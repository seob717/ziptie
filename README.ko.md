*"Rules with nunchi — delivered before you have to ask."*

nunchi는 CLAUDE.md 규칙을 이벤트 리스너로 바꾸는 Claude Code 플러그인입니다. 세션 시작에 모든 규칙을 실어 두고 40턴과 컴팩션을 버티기를 바라는 대신, 각 규칙을 해당 행동이 발생하는 바로 그 순간에 배달합니다:

![실제 세션: Claude가 버그를 고치고 gh pr create를 실행하자 nunchi가 첫 시도를 한 번 차단하며 docs/pr-rules.md를 사유로 배달하고, 규칙이 컨텍스트에 실린 재시도는 통과한다](assets/nunchi-demo.gif)

*(실제 세션 리플레이 — 첫 `gh pr create`는 정확히 한 번 거부되고, 그 순간 `docs/pr-rules.md`에서 읽어 온 규칙 문서 전문이 거부 사유로 배달되며, 재시도는 통과합니다)*

`/nunchi:compile` 한 번이 CLAUDE.md를 트리거가 바인딩된 룰 파일로 바꾸고, 그 뒤로는 PreToolUse 훅이 배달을 맡습니다. 얻는 것은 다음과 같고, 전부 [실측](#실측-결과)입니다:

- **세션 시작 토큰 −42.5%** — 룰 문서 8개(~76KB)를 `@import`에서 빼자 세션 시작마다 프롬프트 토큰 ~34k가 절약됐습니다. 문서 전문의 비용은 해당 행동이 실제로 발생하는 세션에서만 지불합니다.
- **규칙이 컴팩션을 살아남음** — SessionStart 훅이 매 `/compact` 후 배달된 규칙을 재무장하며, 반복 컴팩션에 걸쳐 검증됐습니다.
- **모든 배달이 기록됨** — `/nunchi:report`가 어떤 규칙이 언제 발동했고 당신의 레포에서 얼마를 절약했는지 보여 줍니다.

## 빠른 시작

플러그인을 지원하는 [Claude Code](https://code.claude.com)와 Python 3가 필요합니다(훅 엔진은 표준 라이브러리만으로 동작 — 외부 패키지 없음).

```
/plugin marketplace add seob717/nunchi
/plugin install nunchi@nunchi-marketplace
```

그다음 프로젝트 안에서:

1. **`/nunchi:compile`** — CLAUDE.md와 그 안에서 `@참조`된 문서를 읽어 규칙을 추출하고, 각 규칙의 트리거(도구, 정규식 패턴)를 추론해 `.claude/rules/*.md`로 씁니다.
2. **룰 파일 검토** — 룰 파일은 평문이며 단일 진실 원천(source of truth)입니다. 트리거·강도·source 경로가 맞는지 확인하고 필요하면 직접 고치세요.
3. **끝.** 이제부터 PreToolUse 훅이 각 규칙을 해당 행동 직전에 배달합니다. `/nunchi:report`가 무엇이 언제 배달됐고 얼마를 절약했는지 보여 줍니다.

## 문제

CLAUDE.md에 쓴 규칙은 세션이 시작하는 t=0에 한 번 로드되고, 그걸로 끝입니다. 하지만 그 규칙이 실제로 중요해지는 순간은 대개 수십 턴 뒤입니다. 컨텍스트가 쌓일수록 세션 맨 위에 있던 규칙에 대한 모델의 주의는 옅어지고, 컴팩션(요약)이 한 번 지나가면 명시적 규칙은 흐릿한 배경으로 강등됩니다. `@docs/pr-rules.md` 같은 참조 문서는 규칙이 정작 필요한 시점(예: `gh pr create` 실행)에 컨텍스트에서 가장 먼 곳에 있게 됩니다. nunchi는 규칙을 "세션 맨 위의 선언"이 아니라 "행동에 바인딩된 이벤트 리스너"로 컴파일해서, 규칙이 필요한 시점과 배달되는 시점 사이의 거리를 0으로 만듭니다.

## 동작 방식

1. **`/nunchi:compile`** — CLAUDE.md와 그 안에서 `@참조`된 문서를 읽어 규칙을 추출하고, 각 규칙의 트리거(도구, 정규식 패턴)를 추론해 `.claude/rules/*.md`로 컴파일합니다.
2. **룰 파일 검토** — 컴파일 산출물은 읽고 편집할 수 있는 평문 파일입니다. 트리거·강도·source 문서 경로가 맞는지 확인하고 필요하면 직접 고치세요. 이 파일들이 단일 진실 원천입니다.
3. **행동 시점의 JIT 배달** — PreToolUse 훅이 도구 호출을 가로채 트리거와 대조합니다. 매칭되면 그 순간 `source` 문서를 직접 읽어(붙여 넣은 사본이 아니라 원본을) 배달합니다. 여러 규칙이 같은 도구 호출에 매칭되면 내용이 한 블록으로 함께 배달되므로, 한 행동의 비용은 최대 재시도 한 번입니다.
4. **컴팩션 후 재무장** — 컴팩션에 스코프된 SessionStart 훅이 세션의 배달 마커를 리셋합니다. 컴팩션 전에 이미 배달됐던(그래서 본문이 요약으로 날아간) 규칙도, 다음에 트리거가 매칭될 때 다시 JIT로 배달됩니다. 컴팩션 시점에는 컨텍스트를 전혀 쓰지 않습니다.

## 룰 파일 형식

```markdown
---
name: pr-rules
trigger:
  tool: Bash
  pattern: gh\s+pr\s+create
source: docs/pr-rules.md
strength: require-read
enabled: true
---
Reflect docs/pr-rules.md before creating a PR.
```

- `name`: 소문자·숫자·하이픈만 허용(`[a-z0-9][a-z0-9-]*`). 이름이 규격에 어긋나는 파일은 조용히 오동작하는 대신 stderr 경고와 함께 거부됩니다.
- `trigger.tool` / `trigger.pattern`: 어떤 도구 호출을 어떤 정규식으로 가로챌지. 기본적으로 Bash 룰은 명령 문자열에, Edit/Write 룰은 파일 경로에 매칭됩니다.
- `trigger.field` (선택): 기본값 대신 특정 `tool_input` 필드에 매칭합니다. Edit 룰에 `field: new_string`을 주면 내용 규칙을 표현할 수 있습니다 — 예컨대 패턴 `console\.log`는 파일 경로와 무관하게 *지금 쓰이는 편집 내용*에 `console.log`가 들어 있을 때 발동합니다.
- `trigger.path` (선택): `pattern`과 AND로 결합되는 파일 경로 정규식 — 둘 다 매칭될 때만 발동합니다. 내용 규칙은 코드 파일로 스코프하세요(`path: \.(ts|tsx)$`). 그래야 마크다운·문서 안의 예시 코드가 트리거되지 않습니다. 이게 없으면 `field: new_string` 룰은 편집 내용에 패턴이 들어 있는 *모든* 파일을 차단합니다 — 그 룰의 source 문서 자신까지 포함해서요.
- `source`: 배달 시점에 그 자리에서 읽는 원본 문서 경로. 원본이 바뀌면 다음 배달부터 자동 반영됩니다. 여러 규칙이 하나의 source를 공유하면 문서 전문은 **세션당 한 번만** 이동합니다 — 이후 배달은 문서를 반복하는 대신 룰의 한 줄 본문과 트랜스크립트에 이미 실린 전문을 가리키는 포인터를 싣습니다.
- `strength`: 강제력이 강한 순서대로 세 단계입니다.
  - `block` — 항상 차단하며 규칙을 사유로 배달합니다. 문서가 절대 금지로 명시한 행동용. 같은 세션의 반복 시도도 계속 차단되지만, 사유는 축약본(한 줄 본문 + 앞서 배달된 전문 포인터)으로 갑니다 — 강제력은 그대로, 토큰은 절약.
  - `require-read` (기본값) — 세션당 한 번, 규칙 본문을 사유로 차단한 뒤 재시도는 통과시킵니다. 재시도 한 번으로 확실한 읽기를 삽니다.
  - `inject` — 도구 호출과 함께 `additionalContext`로 규칙을 배달하며, **차단도 재시도 비용도 0**입니다. 배달에는 출처 프레이밍(프로젝트 소유자의 훅, 등록된 룰 경로, `source` 경로)이 실립니다 — 프레이밍 없는 주입 지시는 프롬프트 인젝션으로 취급돼 거부되고, 프레이밍된 것은 준수된다는 것을 실측했기 때문입니다(`pilot/PROBE-inject.md` 참조). `require-read`보다 부드럽습니다: 준수가 강제 재시도가 아니라 모델의 판단에 실립니다. nunchi는 `permissionDecision: allow`를 절대 반환하지 않으므로 권한 프롬프트는 건드리지 않습니다.
- 본문(body): 원본 문서 대신, 또는 그에 더해 배달할 요약. **한 줄로 유지하세요** — 아래 참조.

### Claude Code 네이티브 `.claude/rules/` 로더와의 상호작용

Claude Code 자체(v2.0.64+)도 `.claude/rules/*.md`를 읽습니다: `paths:` frontmatter 키가 없는 파일은 본문이 세션 시작에 컨텍스트로 로드됩니다(v2.1.206에서 프로브로 검증). nunchi는 이와 싸우는 대신 껴안습니다 — 두 로더가 일을 나눕니다:

- **본문**은 t=0에 네이티브로 로드되는 상시 한 줄 선언을 겸합니다("이 레포에는 PR 규칙이 있다").
- **source 문서**는 트리거가 발동하는 순간 nunchi가 전문으로 JIT 배달하는 대상입니다.

룰 본문을 요약 한 줄로 유지해야 하는 이유가 이것입니다: 본문이 길면 세션 시작에 주입되고 *또* 트리거 시점에 다시 배달됩니다. `/nunchi:compile`은 기본적으로 본문을 이렇게 생성합니다.

### 컴파일된 룰을 신선하게 유지하기

룰 *내용*의 변경에는 아무것도 할 필요가 없습니다 — `source` 문서는 매 배달마다 새로 읽힙니다. 조용히 낡을 수 있는 것은 *트리거 구조*입니다: 문서에 새 규칙이 추가되거나, 규칙이 다른 행동에 재바인딩되거나, 금지 강도가 바뀌는 경우 — 어느 것도 `/nunchi:compile <path>`를 다시 돌리기 전에는 반영되지 않습니다. 세 가지 신호가 이를 커버합니다([#17](https://github.com/seob717/nunchi/issues/17)):

- **편집 시점 알림** — 도구 호출이 컴파일된 룰의 `source`인 문서를 편집하면, 훅이 영향받는 룰 이름과 정확한 재컴파일 명령을 담은 세션당 1회 안내를 주입합니다. 비차단(`additionalContext`만)이며, 매칭용으로 이미 로드된 룰 목록을 재사용하므로 아무것도 건드리지 않는 호출은 비용이 없습니다.
- **`/nunchi:report` 후보** — `source` 문서가 디스크에서 컴파일된 룰 파일보다 최신인 룰은 재컴파일 후보로 표시됩니다(mtime 휴리스틱 — 새로 클론하면 mtime이 리셋되므로 증거가 아니라 확인 요청으로 받아들이세요).
- **컨벤션 워처(옵트인)** — 룰 문서가 `docs/*-rules.md` 같은 경로 컨벤션을 따른다면, 평범한 nunchi 룰 하나가 컨벤션 자체를 감시하며 새로 생긴 문서의 컴파일을 제안할 수 있습니다(`/nunchi:compile`이 생성을 제안 — compile 명령의 §5.6 참조). 별도 설정 표면은 없습니다 — 워처도 그저 룰 파일 하나입니다.

이 셋 모두 못 잡는 유일한 틈은 문서 변경이 전혀 없는 트리거 드리프트입니다(팀이 `gh` → `glab`으로 갈아탔는데 PR 규칙 문서는 도구를 언급조차 안 하는 경우) — 이는 `/nunchi:report`에서 죽은 룰(배달 0회)로 드러나며, 리포트가 한 번도 트리거되지 않은 룰을 표시하는 이유가 그것입니다.

## 실측 결과

측정 하니스: 샌드박스 레포 + 목(mock) `gh`(PR 캡처) + 기계 채점 가능한 PR 규칙 4개 + 헤드리스 `claude -p`(sonnet), 조건당 3회 실행. (하니스 전체가 `pilot/` 디렉터리에 보존돼 있어 재실행할 수 있습니다.)

| 조건 | 설명 | 전부 통과 | 비고 |
|---|---|---|---|
| A | CLAUDE.md만 (짧은 컨텍스트) | 3/3 | 천장 효과 — 규칙이 방금 로드돼서 이 압력에선 무너지지 않음 |
| AL | CLAUDE.md만 + 긴 컨텍스트 (~60k 토큰) | 3/3 | 긴 컨텍스트에서도 천장 효과 지속 |
| HW | hookify warn | 3/3 | 경고 메시지가 모델에 도달하지 않아 사실상 A와 같은 조건 |
| HB | hookify block | 0/3 | `permissionDecisionReason`을 채우지 않아 모델이 이유도 모른 채 차단되고, 과제 3/3이 파괴됨 |
| Z | nunchi JIT 배달 (실제 엔진, E2E) | 3/3 | 훅 발동 3/3 — 첫 시도는 차단되며 사유(규칙 원문)가 배달되고, 재시도는 모두 통과 |

이 표가 보여 주는 것은 "JIT가 CLAUDE.md보다 준수율이 높다"는 우위가 *아닙니다* — 이 파일럿의 압력 수준(단순 규칙 4개, 단일 과제, sonnet)에서는 CLAUDE.md만으로도(A, AL) 천장 효과로 버텼고, 우리는 그 결과를 숨기지 않습니다. 측정이 실제로 뒷받침하는 것은 세 가지입니다:

1. **hookify block은 해롭다.** 이유를 전달하지 않는 차단은 과제 3/3을 파괴합니다. 같은 차단이라도 사유를 함께 배달하면(nunchi 방식) 0/3 → 3/3으로 뒤집힙니다.
2. **JIT 배달 메커니즘 자체가 동작하며 과제를 해치지 않는다.** 실제 nunchi 엔진을 붙인 E2E(조건 Z)에서 훅이 3/3 정상 발동했고, 차단 → 사유 전달 → 재시도를 거친 3회 실행 모두 과제를 완료했습니다.
3. **source 문서 동기화와 배달 로깅이 실제로 동작한다.** `source`는 배달 때마다 매번 읽히므로 규칙과 원본이 어긋날 수 없고, 모든 트리거·배달·차단이 JSONL로 기록돼 `/nunchi:report`가 집계합니다.

**압력 재검증(사전 등록):** 이후 압력을 3단계 `@참조` 구조 뒤의 규칙 24개 + 동일한 ~268KB 장문 컨텍스트 과제로 키워 CLAUDE.md-단독 조건을 재실행했습니다(설계와 결과: `pilot/DESIGN-pressure.md`, `pilot/RESULTS-pressure.md`). 천장은 유지됐습니다 — 유효 5회 실행(채점 체크포인트 40/40)에서 전부-통과 100%. 따라서 확증적 JIT-대-CLAUDE.md 비교의 사전 등록 게이트가 충족되지 않았고, 비교는 실행하지 않았습니다.

**컴팩션 압력(사전 등록):** 이어서 과제 중간에 `/compact`를 강제하고 컴팩션 이후 행동만 채점했습니다(설계와 결과: `pilot/DESIGN-compaction.md`, `pilot/RESULTS-compaction.md`). CLAUDE.md-단독 조건에서 전체 실험을 통틀어 첫 규칙 위반이 관측됐지만(`@참조` 문서에만 있는 허용 목록 밖 값으로 라벨링된 PR — 정확히 컴팩션이 버리는 종류의 디테일), 전부-통과 2/3으로 사전 등록 게이트는 이번에도 미충족이라 확증 비교는 실행하지 않았고 우위 주장도 하지 않습니다. 실행이 뒷받침하는 것: 재무장 메커니즘은 처치군 3/3에서 동작했고(배달 → 컴팩션 → 재무장 → 재배달), JIT 배달은 컴팩션 후에도 과제 완료를 해치지 않았습니다(전부-통과 3/3). 더 강한 압력에서 JIT가 준수 우위를 보이는지는 미검증으로 남아 있고, 검증되기 전까지 그 주장은 하지 않습니다.

**컴팩션 후속(사전 등록):** 추가 2개 암(설계와 결과: `pilot/DESIGN-compaction-followup.md`, `pilot/RESULTS-compaction-followup.md`). CLAUDE.md-단독 컴팩션 조건의 10회 확장 실행에서 유효 9회에 새 위반이 없어, 관측된 단일-컴팩션 위반율은 1/12(~8%)입니다. 이중-컴팩션 예비 실행(3단계 과제, `/compact` 2회 강제)도 사전 등록 게이트를 충족하지 못해(유효 베이스라인 실행 전부 통과) 우위 비교는 실행하지 않았고 주장하지 않습니다. 검증된 것: **재무장 메커니즘은 반복 컴팩션을 살아남습니다** — 배달 → 컴팩션 → 재배달 → 컴팩션 → 재배달, 유효 처치 2/2 — 그리고 JIT 배달은 컴팩션 두 번 후에도 과제 완료를 해치지 않았습니다.

**컨텍스트 경제성(실측 프로브):** `@import`된 파일은 "실행 시 확장되어 컨텍스트에 로드"되며([공식 메모리 문서](https://code.claude.com/docs/en/memory)), 공식 모범 사례는 "파일이 길수록 컨텍스트를 더 소모하고 준수율을 낮춘다"고 경고합니다([best practices](https://code.claude.com/docs/en/best-practices)) — 공식 처방인 경로 스코프 룰은 파일 읽기 트리거를 커버하고, nunchi는 같은 아이디어를 강제력을 갖춘 행동 트리거로 확장합니다. 룰 문서를 CLAUDE.md `@import`에서 nunchi 룰로 옮기면 그 토큰을 지불하는 *시점*이 바뀝니다. 이를 정면 비교로 측정했습니다(`pilot/PROBE-context-economics.md`): 같은 레포에서 룰 문서 8개(~76KB)를 `@import`로 로드하면 **세션 시작에 프롬프트 토큰 79,683**, nunchi 구성(한 줄 룰 본문, 문서는 트리거로 JIT 배달)은 **45,808** — **세션 시작당 ~34k 토큰(−42.5%) 절약**, 룰 문서 페이로드 자체는 97% 감소(34,878 → 1,003 토큰)입니다. 문서 전문의 비용은 해당 행동이 실제로 발동하는 세션에서, 발동하는 순간에만 지불합니다. 정직한 캐비앗 둘: 상시 지침도 겸하는 문서(아래 컴파일 벤치마크 기준 야생 규칙의 약 절반)는 `@참조`를 유지해야 합니다 — 절약은 행동에 바인딩 가능한 문서에 적용됩니다. 그리고 많은 규칙이 트리거되는 세션에서는 배달 지출이 도로 쌓입니다 — 다만 선불 화물이 아니라 정확한 시점에 도착하는 컨텍스트로요. `/nunchi:report`가 당신의 레포에서 이 숫자를 룰 파일과 배달 로그로부터 계산해 줍니다.

**야생 CLAUDE.md 컴파일 벤치마크(사전 등록 파일럿):** `/nunchi:compile` 지침을 실제 CLAUDE.md 파일 12개(총 166KB: apache/airflow, vercel/next.js, supabase, 그리고 [awesome-claude-md](https://github.com/josix/awesome-claude-md)의 결정론적 샘플, 빈 스텁 음성 대조군 포함) 위에서 실행했습니다 — 설계와 결과: `pilot/DESIGN-compile-bench.md`, `pilot/RESULTS-compile-bench.md`. 추출된 규칙 88개 중: 정규식·도구 유효성 100%, 내용 규칙의 필수 `path:` 스코프 보유율 100%, 빈 대조군에서 날조된 규칙 0, 그리고 `block` 강도 배정 8건 전부가 원문의 명시적 "Never/MUST NOT" 문구로 소급됐습니다(인용 단위로 검증). 야생 규칙의 약 절반(51%)만 행동에 바인딩 가능했습니다 — 나머지 절반은 상시 지침이고, 이것이 바로 nunchi가 CLAUDE.md를 대체하는 게 아니라 공존하도록 설계된 이유입니다. 적대적 골드셋(격리 세션의 별도 모델이 작성한 규칙 181개) 대비 재현율은 **마이크로 재현율 35%** — 숨기지 않고 측정했습니다(`pilot/RESULTS-compile-recall.md`, `pilot/RESULTS-compile-recall-v2.md`). 그 미스의 약 3분의 1은 컴파일러가 행동에 바인딩 불가능한 지침을 *의도적으로* 건너뛴 것이고(산출물에 자가 보고됨), 골드셋은 스펙의 트리거당-한-규칙 정책보다 규칙을 잘게 쪼갭니다 — 그러나 실제 공백도 남아 있고(표와 깊은 목록 안의 규칙), 이를 좁히는 작업은 이슈로 공개 추적 중입니다.

**다국어 컴파일 호환성(사전 등록 파일럿):** 같은 컴파일 지침을 위 문서 중 3개의 한국어/일본어/스페인어 번역본 위에서 돌렸을 때 재현율·출력 형식·과추출 모두 영어 원본 대비 저하가 없었습니다(`pilot/RESULTS-compile-multilingual.md`) — 언어에 따라 흔들린 유일한 축은 **강도**였습니다: 권고 대 금지의 뉘앙스("avoid" 대 "never")가 번역에서 왜곡됩니다. 컴파일 스펙에 화행(speech-act) 판정 원칙과 언어별 보정 예시를 추가하자 골드 강도 일치율이 61%에서 82%로 올랐고, 게이트 지표에는 회귀가 없었습니다(`pilot/RESULTS-strength-guidance.md`). 이어서 같은 파이프라인을 GitHub에서 결정론적으로 샘플링한 *야생* 한국어 CLAUDE.md 4개(pinpoint-apm/pinpoint, ★13.8k 포함)에 돌렸습니다: 형식 위반 0, 과추출 0, 날조 증거 0, 강도 일치율 88% — "master에는 절대 직접 커밋하지 않습니다" 같은 야생 금지문은 모두 `block`으로 컴파일됐습니다(`pilot/RESULTS-compile-wild-ko.md`). 한국어로 CLAUDE.md를 쓰는 저자를 위한 문구 가이드가 있습니다: [`guides/writing-korean-claude-md.md`](guides/writing-korean-claude-md.md).

## FAQ — "이거 그냥 …아닌가요?"

**…CLAUDE.md 다이어트?** CLAUDE.md를 줄이면 선불 토큰 청구서는 줄지만 배달 모델은 그대로입니다: 남긴 것 전부가 여전히 t=0에 로드되고 거리와 컴팩션에 따라 옅어집니다. nunchi는 규칙의 양이 아니라 도착 *시점*을 바꿉니다 — 규칙 본문이 그것이 다스리는 행동 몇 밀리초 전에 컨텍스트에 착지하고, 매 컴팩션 후에도 다시 착지합니다. 어느 쪽이든 실측입니다: 관측된 베이스라인 실패가 정확히 `@참조` 문서에 있던 규칙의 컴팩션-후 위반이었습니다(`pilot/RESULTS-compaction.md`).

**…Claude Code의 경로 스코프 룰(`paths:` frontmatter)?** 경로 스코프 룰은 *파일 읽기*에 트리거됩니다. nunchi는 *행동*에 트리거됩니다: `gh pr create`, `git commit`, 지금 쓰이는 편집의 내용(`trigger.field: new_string`) — 세션별 배달 상태와 3단계 강제력까지 갖추고요. 둘은 결합됩니다 — nunchi의 룰 파일은 네이티브 `.claude/rules/` 파일이고, 네이티브 로더가 한 줄 본문을 세션 시작에 주입하는 동안 nunchi는 트리거 시점에 source 문서 전문을 배달합니다.

**…스킬?** 스킬과 nunchi는 같은 컨텍스트 형태를 공유합니다 — 항상 로드되는 한 줄 선언, JIT로 오는 본문 전문 — 하지만 트리거의 주인이 다릅니다. 스킬은 *풀(pull)*입니다: 모델이 설명을 보고 필요함을 스스로 알아채 열어야 하고, 특정 행동 전에 스킬이 발동하도록 강제하는 메커니즘은 없습니다([공식 문서](https://code.claude.com/docs/en/skills)도 보장이 필요하면 훅을 쓰라고 안내합니다). 규칙 준수는 정확히 모델이 규칙의 존재를 잊었을 때 실패합니다 — 그리고 규칙을 잊은 모델은 그 스킬도 열지 않습니다. nunchi는 *푸시(push)*입니다: 행동 자체가 트리거라서 배달이 모델의 기억에 의존하지 않고, 강제할 수 있으며(`require-read`/`block` — 스킬은 권고 전용), 모든 배달이 기록됩니다. 요약하면: **스킬은 모델이 기억할 때 동작하고, nunchi는 잊었을 때 동작합니다.** 둘은 결합됩니다 — 절차(릴리스 단계, 마이그레이션 워크플로)는 스킬에 두고, 그 절차가 취하는 행동은 nunchi가 지키게 하세요: 절차 중간에 `gh pr create`를 실행하는 스킬은 바로 그 자리에서 PR 규칙을 배달받습니다.

**…Context Mode나 RTK 같은 MCP/출력 압축 도구?** 그것들은 컨텍스트로 *들어오는* 것(MCP 출력, Bash 출력)을 압축합니다. nunchi는 아무것도 압축하지 않습니다 — 규칙 배달을 스케줄링합니다. 토큰 절약은 문서를 선적재하지 않은 부수 효과이고, 제품은 행동의 순간에 규칙이 컨텍스트에 있음을 증명 가능하게 만드는 것, 그리고 모든 배달이 기록돼 준수를 가정하는 대신 감사할 수 있게 하는 것입니다.

**…정규식이 맞기를 비는 것?** 트리거는 설계상 결정론적 정규식입니다 — 감사 가능하고, 테스트 가능하고, 검색 분산이 없습니다. 비용은 정직합니다: 잘못 스코프된 패턴은 놓치거나 과발동합니다. 그래서 컴파일된 룰 파일이 사람이 검토하는 평문이고(빠른 시작 2단계), 내용 규칙에 `path:` 스코프가 필수이며, 컴파일 스펙 자체가 야생 코퍼스에서 형식 유효성(100%)과 날조(0)로 벤치마크되는 것입니다.

## 다른 도구와의 비교

각 도구는 "에이전트가 규칙을 따르게 하기"의 서로 다른 층을 풉니다. 정적 배포기([Ruler](https://github.com/intellectronica/ruler), [rulesync](https://github.com/dyoshikawa/rulesync))는 *"같은 규칙을 모든 에이전트에 어떻게 배포하나?"*에 답합니다. 가드레일 훅([hookify](https://github.com/anthropics/claude-code/tree/main/plugins/hookify))은 *"나쁜 행동을 어떻게 막나?"*에 답합니다. [스킬](https://code.claude.com/docs/en/skills)은 *"선불 비용 없이 에이전트에 절차를 어떻게 주나?"*에 답합니다. nunchi는 *"필요한 순간에 맞는 규칙을 모델 컨텍스트에 어떻게 넣고 — 컴팩션을 넘어 어떻게 유지하나?"*에 답합니다. 이들은 결합됩니다: 규칙 문서는 Ruler로 배포하고, 절차는 스킬에 두고, 규칙 배달은 nunchi가 맡을 수 있습니다.

아래 표는 **점수판이 아니라 범위 지도**입니다 — 빈 칸이나 "by design" 표기는 대개 의도된 설계 선택이고, 여러 행은 반대 방향입니다(hookify는 더 많은 훅 이벤트를 커버하고, Writ는 의미 기반으로 검색하고, Ruler는 30+ 에이전트에 배포합니다 — 셋 다 nunchi는 하지 않습니다).

훅 오버헤드 실측: 도구 호출당 중앙값 ~24ms (규칙이 배달될 때 26ms). 각 행은 2026년 7월 기준 각 도구의 소스 또는 공식 문서로 검증했습니다 — 정정은 이슈로 환영합니다.

공로는 공로대로: hookify는 Claude Code 훅 위의 마크다운 frontmatter 룰을 개척했고 더 많은 훅 이벤트(PostToolUse, Stop, UserPromptSubmit)를 커버합니다. 스킬은 절차의 올바른 거처입니다 — 스킬당 ~100 토큰의 점진적 공개에 nunchi에는 없는 실행 제어(`allowed-tools`, 모델/노력 오버라이드, 포크된 컨텍스트)까지. Ruler와 rulesync는 크로스 에이전트 배포를 제대로 풉니다. Writ는 의미 기반 검색 배달을 탐구합니다. nunchi의 틈새는 의도적으로 좁습니다: 결정론적·행동 트리거 배달과 배달 상태 추적 — 그리고 오직 실측된 주장만.

## 한계와 로드맵

다음은 아직 구현되지 않았으며 로드맵에 있습니다.

- **의미 기반 판정**: 정규식 트리거가 아니라 LLM으로 출력 내용을 검사해 규칙 위반을 잡는 것. 지연/비용 트레이드오프 검토가 필요합니다.
- **Stop 이벤트 룰**: 세션 종료 시점에 "과제 완료 전에 이 조건이 충족됐는가?"를 검사하는 룰.
- **준수 리포트 UI**: 현재 `/nunchi:report`는 로그를 표로 집계할 뿐이고, 더 정교한 분석은 백로그에 있습니다.

또한 이 결과들은 조건당 n=3 파일럿과 사전 등록된 압력 예비 실행(6회)에서 나왔습니다. 통계적으로 "100%"는 "이 표본에서 실패를 관측하지 못했다" 이상을 의미하지 않으며, JIT 준수 우위 주장은 베이스라인을 실제로 무너뜨리는 압력 수준이 나올 때까지 유보합니다.
