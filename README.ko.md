*"Rules with nunchi — delivered before you have to ask."*

nunchi는 CLAUDE.md 규칙을 이벤트 리스너로 바꿔 주는 Claude Code 플러그인입니다. 규칙 전부를 세션 시작에 실어 두고 40턴의 대화와 컴팩션을 무사히 버티기를 바라는 대신, 각 규칙을 해당 행동이 일어나는 바로 그 순간에 전달합니다:

![실제 세션: Claude가 버그를 고치고 gh pr create를 실행한다. nunchi가 첫 시도를 한 번 차단하면서 docs/pr-rules.md를 사유로 전달하고, 규칙을 받은 재시도는 통과한다](assets/nunchi-demo.gif)

*(실제 세션을 다시 재생한 것입니다 — 첫 `gh pr create`는 정확히 한 번만 거부되고, 그 시점에 `docs/pr-rules.md`에서 읽어 온 규칙 문서 전문이 거부 사유로 함께 전달되며, 재시도는 통과합니다)*

`/nunchi:compile`을 한 번 실행하면 CLAUDE.md가 트리거에 묶인 룰 파일들로 바뀌고, 그다음부터는 PreToolUse 훅이 전달을 맡습니다. 얻는 것은 다음과 같고, 모두 [실측 결과](#실측-결과)입니다:

- **세션 시작 토큰 −42.5%** — 룰 문서 8개(~76KB)를 `@imports`에서 빼는 것만으로 세션을 열 때마다 프롬프트 토큰이 약 34k씩 줄었습니다. 문서 전문의 비용은 그 행동이 실제로 일어나는 세션에서만 냅니다.
- **규칙이 컴팩션 뒤에도 살아남습니다** — SessionStart 훅이 `/compact`가 지나갈 때마다 이미 전달된 규칙을 재활성화합니다. 컴팩션을 반복해도 유지되는 것을 검증했습니다.
- **모든 전달이 기록됩니다** — `/nunchi:report`로 어떤 규칙이 언제 발동했고 내 레포에서 얼마나 절약했는지 확인할 수 있습니다.

## 빠른 시작

플러그인을 지원하는 [Claude Code](https://code.claude.com)와 Python 3만 있으면 됩니다. 훅 엔진은 표준 라이브러리로만 동작하므로 따로 설치할 패키지가 없습니다.

**Option 1 — 플러그인으로 설치** (권장: 스스로 최신을 유지할 수 있는 관리형 번들):

```bash
claude plugin marketplace add seob717/nunchi
claude plugin install nunchi@nunchi-marketplace
```

(세션 안에서는 `/plugin marketplace add seob717/nunchi`와 `/plugin install nunchi@nunchi-marketplace`가 같은 일을 합니다.)

서드파티 마켓플레이스는 자동 업데이트가 **기본으로 꺼져** 있습니다. 릴리스가 나올 때마다 자동으로 받으려면 `/plugin`을 실행해 **Marketplaces** 탭에서 `nunchi-marketplace`를 고르고 **Enable auto-update**를 켜세요. 업데이트는 릴리스 단위로 나갑니다(release-please가 플러그인 버전을 올립니다). Claude Code가 세션 시작 후 백그라운드에서 받아 오고, 다음 실행 또는 `/reload-plugins` 때 적용됩니다. 직접 관리하고 싶다면 `/plugin marketplace update nunchi-marketplace`로 원할 때 갱신하면 됩니다.

팀 전체에 설치하려면 프로젝트의 `.claude/settings.json`에 아래를 커밋하세요 — 저장소를 신뢰한 협업자에게 설치 안내가 뜹니다:

```json
{
  "extraKnownMarketplaces": {
    "nunchi-marketplace": {
      "source": { "source": "github", "repo": "seob717/nunchi" }
    }
  },
  "enabledPlugins": {
    "nunchi@nunchi-marketplace": true
  }
}
```

**Option 2 — 클론해서 고치며 쓰기** (수정 가능한 로컬 사본: 엔진·명령·트리거를 직접 손볼 수 있습니다):

```bash
git clone https://github.com/seob717/nunchi.git
claude --plugin-dir /path/to/nunchi
```

`--plugin-dir`는 호출 단위 지정이라 실행할 때마다 붙여야 합니다. 로컬 수정은 다음 실행부터 적용되고, 밑에서 몰래 바뀌는 일은 없습니다.

그다음 프로젝트 안에서:

1. **`/nunchi:compile`** — CLAUDE.md와 그 안의 `@referenced` 문서를 읽어 규칙을 추출하고, 규칙마다 트리거(도구, 정규식 패턴)를 추론해 `.claude/rules/*.md`로 저장합니다.
2. **룰 파일을 검토합니다** — 룰 파일은 평문이고, 이 파일들이 기준(source of truth)입니다. 트리거·강제 수준·source 경로가 맞는지 확인하고, 틀렸으면 직접 고치면 됩니다.
3. **끝입니다.** 이후로는 PreToolUse 훅이 각 규칙을 그 규칙이 적용될 행동 직전에 전달합니다. 무엇이 언제 전달됐고 얼마나 절약했는지는 `/nunchi:report`로 봅니다.

## 문제

CLAUDE.md에 적은 규칙은 세션이 시작되는 t=0에 한 번 로드되고 끝입니다. 정작 그 규칙이 중요해지는 순간은 대개 수십 턴 뒤에 옵니다. 컨텍스트가 쌓일수록 세션 첫머리의 규칙에 대한 모델의 주의는 흐려지고, 컴팩션(요약)을 한 번 거치면 명시적이던 규칙도 흐릿한 배경 지식으로 밀려납니다. `@docs/pr-rules.md`처럼 참조로 걸어 둔 문서는 규칙이 가장 필요한 순간(예: `gh pr create`를 실행할 때) 컨텍스트에서 가장 멀리 있게 됩니다. nunchi는 규칙을 "세션 첫머리의 선언"이 아니라 "행동에 묶인 이벤트 리스너"로 컴파일합니다. 규칙이 필요한 시점과 전달되는 시점 사이의 거리를 없애는 것입니다.

## 동작 방식

1. **`/nunchi:compile`** — CLAUDE.md와 그 안의 `@referenced` 문서를 읽어 규칙을 추출하고, 규칙마다 트리거(도구, 정규식 패턴)를 추론해 `.claude/rules/*.md`로 컴파일합니다.
2. **룰 파일을 검토합니다** — 컴파일 결과물은 읽고 고칠 수 있는 평문 파일입니다. 트리거·강제 수준·source 문서 경로가 맞는지 확인하고, 틀렸으면 직접 고치면 됩니다. 이 파일들이 기준(source of truth)입니다.
3. **행동 시점의 JIT(just-in-time) 전달** — PreToolUse 훅이 도구 호출을 가로채 트리거와 대조합니다. 매칭되면 그 자리에서 `source` 문서를 직접 읽어(복사본이 아니라 원본을) 전달합니다. 한 도구 호출에 여러 규칙이 걸리면 내용을 한 블록으로 묶어 함께 전달하므로, 행동 하나에 드는 비용은 많아야 재시도 한 번입니다.
4. **컴팩션 후 재활성화** — 컴팩션에만 반응하는 SessionStart 훅이 그 세션의 전달 마커를 초기화합니다. 컴팩션 전에 이미 전달됐던(그래서 본문이 요약에 쓸려 나간) 규칙도, 다음에 트리거가 걸리면 다시 JIT로 전달됩니다. 컴팩션 시점에는 컨텍스트를 전혀 쓰지 않습니다.

## 룰 파일 형식

컴파일의 처음과 끝 — CLAUDE.md의 이 한 줄이:

```markdown
All PRs follow the checklist in @docs/pr-rules.md.
```

행동에 묶인 `.claude/rules/pr-rules.md`가 됩니다:

```markdown
---
name: pr-rules
trigger:
  tool: Bash
  pattern: gh\s+pr\s+create
source: docs/pr-rules.md
strength: require-read
enabled: true
---
Reflect docs/pr-rules.md before creating a PR.
```

- `name`: 소문자·숫자·하이픈만 씁니다(`[a-z0-9][a-z0-9-]*`). 규격에 어긋나는 이름의 파일은 조용히 오동작하는 대신 stderr 경고와 함께 거부됩니다.
- `trigger.tool` / `trigger.pattern`: 어떤 도구 호출을 어떤 정규식으로 가로챌지 정합니다. 기본값으로 Bash 룰은 명령 문자열에, Edit/Write 룰은 파일 경로에 매칭합니다.
- `trigger.field` (선택): 기본 대상 대신 `tool_input`의 특정 필드에 매칭합니다. Edit 룰에 `field: new_string`을 주면 내용 규칙을 만들 수 있습니다 — 예를 들어 패턴 `console\.log`는 파일 경로와 상관없이, 지금 쓰려는 편집 내용에 `console.log`가 들어 있으면 발동합니다.
- `trigger.path` (선택): `pattern`과 AND로 묶이는 파일 경로 정규식입니다 — 둘 다 맞을 때만 발동합니다. 내용 규칙은 코드 파일로 범위를 좁히세요(`path: \.(ts|tsx)$`). 그래야 마크다운 문서 안의 예시 코드에는 발동하지 않습니다. 이게 없으면 `field: new_string` 룰은 편집 내용에 패턴이 든 모든 파일을 차단합니다 — 그 룰의 source 문서 자신까지도요.
- `source`: 전달 시점에 그 자리에서 읽는 원본 문서의 경로입니다. 원본이 바뀌면 다음 전달부터 자동으로 반영됩니다. 여러 규칙이 source 하나를 공유하면 문서 전문은 **세션당 한 번만** 나갑니다 — 그다음부터는 문서를 반복하는 대신, 룰의 한 줄 본문과 트랜스크립트에 이미 실린 전문을 가리키는 안내만 전달합니다.
- `strength`: 강제 수준입니다. 강한 순서대로 세 가지가 있습니다.
  - `block` — 항상 차단하고 규칙을 사유로 전달합니다. 문서가 절대 금지라고 못박은 행동에 씁니다. 같은 세션에서 다시 시도해도 계속 차단되지만, 그때의 사유는 축약본(한 줄 본문 + 앞서 전달한 전문 안내)입니다 — 강제력은 그대로 두고 토큰만 아낍니다.
  - `require-read` (기본값) — 세션당 한 번, 규칙 본문을 사유로 차단한 뒤 재시도는 통과시킵니다. 재시도 한 번으로 확실한 읽기를 보장받는 셈입니다.
  - `inject` — 도구 호출과 함께 `additionalContext`로 규칙을 전달합니다. **차단하지 않고 재시도 비용도 없습니다.** 전달할 때는 출처를 밝히는 문구(프로젝트 소유자의 훅이라는 것, 등록된 룰 경로, `source` 경로)를 함께 싣습니다. 출처 없이 주입된 지시는 프롬프트 인젝션으로 취급돼 거부되고, 출처를 밝힌 지시는 준수된다는 것을 실측했기 때문입니다(`pilot/PROBE-inject.md` 참조). `require-read`보다 부드러운 수준으로, 강제 재시도 대신 모델의 판단에 준수를 맡깁니다. nunchi는 `permissionDecision: allow`를 절대 반환하지 않으므로, 권한 프롬프트는 그대로 유지됩니다.
- 본문(body): 원본 문서 대신, 혹은 그에 더해 전달할 요약입니다. **반드시 한 줄로 유지하세요** — 이유는 아래에 있습니다.

### Claude Code 네이티브 `.claude/rules/` 로더와의 상호작용

Claude Code(v2.0.64+) 자체도 `.claude/rules/*.md`를 읽습니다. `paths:` frontmatter 키가 없는 파일은 본문이 세션 시작에 컨텍스트로 로드됩니다(v2.1.206에서 프로브로 확인). nunchi는 이 동작과 싸우는 대신 활용합니다 — 두 로더가 역할을 나눠 갖습니다:

- **본문**은 t=0에 네이티브로 로드되는 상시 한 줄 선언 역할을 합니다("이 레포에는 PR 규칙이 있다").
- **source 문서**는 트리거가 발동하는 순간 nunchi가 전문으로 전달합니다.

룰 본문을 요약 한 줄로 유지해야 하는 이유가 여기에 있습니다. 본문이 길면 세션 시작에 한 번, 트리거 시점에 또 한 번 실리기 때문입니다. `/nunchi:compile`은 기본적으로 본문을 이 형태로 생성합니다.

### 컴파일된 룰을 최신으로 유지하기

룰의 *내용*이 바뀔 때는 할 일이 없습니다 — `source` 문서는 전달할 때마다 새로 읽히니까요. 조용히 낡는 것은 *트리거 구조* 쪽입니다. 문서에 규칙이 새로 생기거나, 규칙이 다른 행동으로 옮겨 가거나, 금지의 강제 수준이 바뀌는 경우인데, 어느 것도 `/nunchi:compile <path>`를 다시 돌리기 전에는 반영되지 않습니다. 세 가지 신호가 이 틈을 메웁니다([#17](https://github.com/seob717/nunchi/issues/17)):

- **편집 시점 알림** — 컴파일된 룰의 `source`인 문서를 편집하는 도구 호출이 오면, 훅이 영향받는 룰 이름과 정확한 재컴파일 명령을 담은 안내를 세션당 한 번 주입합니다. 차단하지 않고(`additionalContext`만 사용), 매칭용으로 이미 읽어 둔 룰 목록을 재사용하므로 해당 없는 호출에는 비용이 들지 않습니다.
- **`/nunchi:report`의 재컴파일 후보** — `source` 문서가 디스크에서 컴파일된 룰 파일보다 최신인 룰을 재컴파일 후보로 보여 줍니다(mtime 기반 추정입니다 — 새로 클론하면 mtime이 초기화되니, 증거가 아니라 확인해 보라는 신호로 받아들이세요).
- **컨벤션 감시(옵트인)** — 룰 문서가 `docs/*-rules.md` 같은 경로 규칙을 따른다면, 평범한 nunchi 룰 하나로 그 경로 규칙 자체를 감시하다가 새로 생긴 문서의 컴파일을 제안하게 할 수 있습니다(`/nunchi:compile`이 생성을 제안합니다 — compile 명령의 §5.6 참조). 설정이 따로 늘어나는 것이 아니라, 감시자도 그냥 룰 파일 하나입니다.

셋 다 못 잡는 유일한 경우는 문서가 전혀 바뀌지 않은 채 일어나는 트리거 드리프트입니다(팀이 `gh`에서 `glab`으로 갈아탔는데 PR 규칙 문서에는 도구 이름이 아예 없는 경우). 이런 룰은 `/nunchi:report`에서 죽은 룰(전달 0회)로 드러납니다. 리포트가 한 번도 발동하지 않은 룰을 따로 표시하는 이유입니다.

## 커맨드 레퍼런스

| 커맨드 | 용도 | 인자 |
|---|---|---|
| `/nunchi:compile` | CLAUDE.md와 그 안의 `@referenced` 문서에서 규칙을 추출해, 트리거에 묶인 `.claude/rules/*.md`를 만듭니다. 건너뛴 내용도 스킬 후보(절차)와 상시 지침(CLAUDE.md에 유지)으로 나눠 보고합니다. | 문서 경로(선택) — 그 문서 하나만 재컴파일합니다(편집 시점 알림이 알려 주는 명령이 바로 이것입니다). |
| `/nunchi:report` | 전달 로그를 표로 집계합니다: 룰별 전달 횟수, 죽은(한 번도 발동하지 않은) 룰, 재컴파일 후보, 그리고 내 레포 기준 컨텍스트 절약 추정. | 없음. |

## 실측 결과

| 조건 | 설명 | 전부 통과 | 비고 |
|---|---|---|---|

| Z | nunchi JIT 전달 (실제 엔진, E2E) | 3/3 | 훅이 3/3 발동 — 첫 시도는 차단되지만 사유(규칙 원문)가 함께 전달되고, 재시도는 모두 통과 |

이 표가 말하는 것은 "JIT가 CLAUDE.md보다 준수율이 높다"는 우위가 *아닙니다*. 이 파일럿의 압력 수준(단순한 규칙 4개, 과제 하나, sonnet)에서는 CLAUDE.md만으로도(A, AL) 천장 효과 덕에 무너지지 않았고, 그 결과를 숨기지 않습니다. 측정이 실제로 뒷받침하는 것은 세 가지입니다:

1. **hookify block은 해롭습니다.** 이유를 알려 주지 않는 차단은 과제 3/3을 망가뜨립니다. 같은 차단이라도 사유를 함께 전달하면(nunchi 방식) 0/3이 3/3으로 뒤집힙니다.
2. **JIT 전달 메커니즘은 동작하고, 과제를 방해하지 않습니다.** 실제 nunchi 엔진을 붙인 E2E(조건 Z)에서 훅이 3/3 정상 발동했고, 차단당한 뒤 사유를 받고 재시도한 3회 모두 과제를 마쳤습니다.
3. **source 문서 동기화와 전달 로그가 실제로 동작합니다.** `source`는 전달할 때마다 새로 읽히므로 규칙과 원본이 어긋날 수 없고, 모든 트리거·전달·차단이 JSONL로 남아 `/nunchi:report`가 집계합니다.

**압력 재검증(사전 등록):** 다음으로 압력을 키웠습니다. 3단계 `@reference` 구조 뒤에 규칙 24개를 두고 같은 ~268KB 장문 컨텍스트 과제를 얹어 CLAUDE.md-단독 조건을 다시 돌렸습니다(설계와 결과: `pilot/DESIGN-pressure.md`, `pilot/RESULTS-pressure.md`). 천장은 그대로였습니다 — 유효 실행 5회에서 전부-통과 100%(채점 체크포인트 40/40). 확증적 JIT 대 CLAUDE.md 비교를 위해 사전 등록해 둔 게이트가 충족되지 않았으므로, 비교는 돌리지 않았습니다.

**컴팩션 압력(사전 등록):** 이번에는 과제 중간에 `/compact`를 강제하고 컴팩션 이후의 행동만 채점했습니다(설계와 결과: `pilot/DESIGN-compaction.md`, `pilot/RESULTS-compaction.md`). CLAUDE.md-단독 조건에서 전체 실험을 통틀어 처음으로 규칙 위반이 관측됐습니다. `@referenced` 문서에만 있는 허용 목록을 벗어난 값으로 PR에 라벨을 단 것인데, 정확히 컴팩션이 버리는 종류의 디테일입니다. 다만 전부-통과 2/3이어서 사전 등록 게이트는 이번에도 미달이었고, 그래서 확증 비교도 돌리지 않았으며 우위 주장도 하지 않습니다. 실행이 뒷받침하는 것은 이렇습니다: 재활성화 메커니즘은 처치군 3/3에서 동작했고(전달 → 컴팩션 → 재활성화 → 재전달), JIT 전달은 컴팩션 후에도 과제 완료를 방해하지 않았습니다(전부-통과 3/3). 더 강한 압력에서 JIT가 준수 우위를 보이는지는 아직 검증되지 않았고, 검증되기 전까지는 그렇게 주장하지 않습니다.

**컴팩션 후속(사전 등록):** 두 갈래를 더 돌렸습니다(설계와 결과: `pilot/DESIGN-compaction-followup.md`, `pilot/RESULTS-compaction-followup.md`). CLAUDE.md-단독 컴팩션 조건을 10회로 늘린 실행에서는 유효 9회에 새 위반이 없었습니다. 이로써 관측된 단일-컴팩션 위반율은 1/12(~8%)입니다. 이중-컴팩션 예비 실행(3단계 과제, `/compact` 2회 강제)도 사전 등록 게이트에 못 미쳐(유효한 베이스라인 실행이 전부 통과) 우위 비교는 돌리지 않았고 주장하지 않습니다. 검증된 것은 이것입니다: **재활성화 메커니즘은 컴팩션을 반복해도 살아남습니다** — 전달 → 컴팩션 → 재전달 → 컴팩션 → 재전달이 유효 처치 2/2에서 확인됐고, JIT 전달은 컴팩션을 두 번 거친 뒤에도 과제 완료를 방해하지 않았습니다.

**컨텍스트 경제성(실측 프로브):** `@imported` 파일은 "실행 시 확장되어 컨텍스트에 로드"되고([공식 메모리 문서](https://code.claude.com/docs/en/memory)), 공식 모범 사례도 "파일이 길수록 컨텍스트를 더 쓰고 준수율이 떨어진다"고 경고합니다([best practices](https://code.claude.com/docs/en/best-practices)). 공식 처방인 경로 스코프 룰은 파일 읽기 트리거까지를 다루고, nunchi는 같은 아이디어를 강제력 있는 행동 트리거로 확장합니다. 룰 문서를 CLAUDE.md의 `@imports`에서 nunchi 룰로 옮기면, 그 토큰을 내는 시점이 달라집니다. 이를 정면 비교로 측정했습니다(`pilot/PROBE-context-economics.md`). 같은 레포에서 룰 문서 8개(~76KB)를 `@import`로 로드하면 세션 시작에 프롬프트 토큰 **79,683**이 들었고, nunchi 구성(한 줄 룰 본문, 문서는 트리거 시점에 JIT 전달)은 **45,808**이 들었습니다. **세션 시작마다 약 34k 토큰(−42.5%)이 절약**되고, 룰 문서 페이로드만 보면 97% 감소입니다(34,878 → 1,003 토큰). 문서 전문의 비용은 그 행동이 실제로 발동하는 세션에서, 발동하는 순간에만 냅니다. 두 가지 한계도 밝혀 둡니다. 첫째, 상시 지침을 겸하는 문서(아래 컴파일 벤치마크 기준으로 야생 규칙의 약 절반)는 `@reference`를 유지해야 합니다 — 이 절약은 행동에 묶을 수 있는 문서에 해당합니다. 둘째, 규칙이 많이 발동하는 세션에서는 전달 비용이 도로 쌓입니다. 다만 세션 시작에 몰아서 내는 대신, 필요한 순간마다 정확히 나눠 내는 구조가 됩니다. `/nunchi:report`가 내 레포의 룰 파일과 전달 로그로 이 숫자들을 계산해 줍니다.

**야생 CLAUDE.md 컴파일 벤치마크(사전 등록 파일럿):** `/nunchi:compile` 지침을 실제 CLAUDE.md 파일 12개 위에서 돌렸습니다(총 166KB: apache/airflow, vercel/next.js, supabase, 그리고 [awesome-claude-md](https://github.com/josix/awesome-claude-md)에서 결정론적으로 뽑은 샘플, 빈 스텁 음성 대조군 포함) — 설계와 결과: `pilot/DESIGN-compile-bench.md`, `pilot/RESULTS-compile-bench.md`. 추출된 규칙 88개의 성적: 정규식·도구 유효성 100%, 내용 규칙의 필수 `path:` 스코프 100%, 빈 대조군에서 지어낸 규칙 0건. `block`으로 배정된 8건은 전부 원문의 명시적 "Never/MUST NOT" 문구로 소급됐습니다(인용 단위로 검증). 야생 규칙 중 행동에 묶을 수 있는 것은 약 절반(51%)뿐이었습니다. 나머지 절반은 상시 지침인데, nunchi가 CLAUDE.md를 대체하지 않고 공존하도록 설계된 이유가 바로 이것입니다. 적대적 골드셋(격리된 세션에서 별도 모델이 만든 규칙 181개) 대비 재현율은 **마이크로 재현율 35%**입니다 — 숨기지 않고 그대로 적습니다(`pilot/RESULTS-compile-recall.md`, `pilot/RESULTS-compile-recall-v2.md`). 놓친 것의 약 3분의 1은 컴파일러가 행동에 묶이지 않는 지침을 *의도적으로* 건너뛴 경우이고(산출물에 스스로 보고합니다), 골드셋이 스펙의 "트리거당 규칙 하나" 정책보다 규칙을 잘게 쪼개는 영향도 있습니다. 그래도 실제 공백은 남아 있습니다(표 안, 깊은 목록 안의 규칙). 좁히는 작업은 이슈로 공개 추적 중입니다.

**다국어 컴파일 호환성(사전 등록 파일럿):** 같은 컴파일 지침을 위 문서 중 3개의 한국어/일본어/스페인어 번역본에 돌렸을 때, 재현율·출력 형식·과추출 어느 것도 영어 원본보다 나빠지지 않았습니다(`pilot/RESULTS-compile-multilingual.md`). 언어에 따라 흔들린 축은 하나, **강제 수준**이었습니다. 권고와 금지의 뉘앙스("avoid"와 "never")가 번역에서 뒤틀립니다. 컴파일 스펙에 화행(speech-act) 판정 원칙과 언어별 보정 예시를 넣자 골드 기준 강제 수준 일치율이 61%에서 82%로 올랐고, 게이트에 걸린 지표들은 그대로였습니다(`pilot/RESULTS-strength-guidance.md`). 이어서 같은 파이프라인을 GitHub에서 결정론적으로 뽑은 *야생* 한국어 CLAUDE.md 4개(pinpoint-apm/pinpoint, ★13.8k 포함)에 돌렸습니다. 형식 위반 0, 과추출 0, 지어낸 근거 0, 강제 수준 일치율 88% — "master에는 절대 직접 커밋하지 않습니다" 같은 야생 금지문은 모두 `block`으로 컴파일됐습니다(`pilot/RESULTS-compile-wild-ko.md`). 한국어로 CLAUDE.md를 쓰는 분을 위한 문구 가이드도 있습니다: [`guides/writing-korean-claude-md.md`](guides/writing-korean-claude-md.md).

## FAQ — "이거 그냥 …아닌가요?"

**…CLAUDE.md 줄이기?** CLAUDE.md를 줄이면 시작 시점의 토큰 청구서는 가벼워지지만, 전달 모델은 그대로입니다. 남겨 둔 것 전부가 여전히 t=0에 로드되고, 거리가 멀어지고 컴팩션이 지나가면 흐려집니다. nunchi가 바꾸는 것은 규칙의 양이 아니라 도착하는 *시점*입니다. 규칙 본문이 그 규칙이 다스리는 행동 직전에 컨텍스트에 도착하고, 컴팩션이 지나간 뒤에도 다시 도착합니다. 어느 쪽이든 측정으로 확인했습니다. 실제로 관측된 베이스라인 실패가 바로 `@referenced` 문서에 있던 규칙의 컴팩션-후 위반이었습니다(`pilot/RESULTS-compaction.md`).

**…Claude Code의 경로 스코프 룰(`paths:` frontmatter)?** 경로 스코프 룰은 *파일을 읽을 때* 발동합니다. nunchi는 *행동*에 발동합니다. `gh pr create`, `git commit`, 지금 쓰려는 편집의 내용(`trigger.field: new_string`)이 트리거이고, 세션별 전달 상태와 세 가지 강제 수준이 있습니다. 둘은 함께 씁니다 — nunchi의 룰 파일은 네이티브 `.claude/rules/` 파일이기도 해서, 네이티브 로더가 한 줄 본문을 세션 시작에 로드하고 nunchi가 트리거 시점에 source 문서 전문을 전달합니다.

**…스킬?** 스킬과 nunchi는 컨텍스트 구조가 같습니다. 한 줄 선언은 항상 로드되고, 본문 전문은 필요할 때 옵니다. 다른 것은 트리거의 주인입니다. 스킬은 *풀(pull)* 방식입니다. 모델이 설명만 보고 지금 필요하다는 것을 스스로 알아채 열어야 하고, 특정 행동 전에 스킬이 반드시 열리게 강제할 방법은 없습니다([공식 문서](https://code.claude.com/docs/en/skills)도 보장이 필요하면 훅을 쓰라고 안내합니다). 규칙 준수가 깨지는 순간은 정확히 모델이 규칙의 존재를 잊었을 때인데, 규칙을 잊은 모델은 그 스킬도 열지 않습니다. nunchi는 *푸시(push)* 방식입니다. 행동 자체가 트리거라서 전달이 모델의 기억에 기대지 않고, 강제할 수 있고(`require-read`/`block` — 스킬은 권고만 가능합니다), 모든 전달이 기록됩니다. 한 줄로 줄이면: **스킬은 모델이 기억할 때 통하고, nunchi는 잊었을 때 통합니다.** 둘은 함께 쓰는 조합입니다. 절차(릴리스 단계, 마이그레이션 순서)는 스킬에 두고, 그 절차가 수행하는 행동은 nunchi가 지키게 하세요. 절차 중간에 `gh pr create`를 실행하는 스킬은 바로 그 자리에서 PR 규칙을 전달받습니다.

**…Context Mode나 RTK 같은 MCP/출력 압축 도구?** 그 도구들은 컨텍스트로 *들어오는* 것(MCP 출력, Bash 출력)을 압축합니다. nunchi는 아무것도 압축하지 않습니다. 규칙 전달의 시점을 잡아 줄 뿐입니다. 토큰 절약은 문서를 미리 싣지 않아서 생기는 부수 효과이고, 핵심 가치는 행동의 순간에 규칙이 컨텍스트에 있음을 증명할 수 있다는 것, 그리고 모든 전달이 기록되어 준수를 가정하는 대신 감사할 수 있다는 것입니다.

**…정규식이 맞기를 비는 것?** 트리거는 일부러 결정론적 정규식으로 만들었습니다. 감사할 수 있고, 테스트할 수 있고, 검색처럼 결과가 흔들리지 않습니다. 대신 비용도 정직합니다. 패턴의 범위를 잘못 잡으면 놓치거나 과하게 발동합니다. 그래서 컴파일된 룰 파일을 사람이 검토하는 평문으로 두고(빠른 시작 2단계), 내용 규칙에는 `path:` 스코프를 필수로 걸고, 컴파일 스펙 자체를 야생 코퍼스에서 형식 유효성(100%)과 날조(0)로 벤치마크합니다.

## 다른 도구와의 비교

"에이전트가 규칙을 따르게 하기"는 층이 여럿인 문제이고, 도구마다 푸는 층이 다릅니다. 정적 배포 도구([Ruler](https://github.com/intellectronica/ruler), [rulesync](https://github.com/dyoshikawa/rulesync))는 *"같은 규칙을 여러 에이전트에 어떻게 배포하나"*를 풉니다. 가드레일 훅([hookify](https://github.com/anthropics/claude-code/tree/main/plugins/hookify))은 *"나쁜 행동을 어떻게 막나"*를 풉니다. [스킬](https://code.claude.com/docs/en/skills)은 *"선불 비용 없이 에이전트에 절차를 어떻게 주나"*를 풉니다. nunchi가 푸는 것은 *"필요한 순간에 맞는 규칙을 모델의 컨텍스트에 넣고, 컴팩션이 지나가도 유지하려면 어떻게 하나"*입니다. 이들은 함께 쓸 수 있습니다. 규칙 문서는 Ruler로 배포하고, 절차는 스킬에 두고, 규칙 전달은 nunchi에 맡기는 식입니다.

아래 표는 **순위표가 아니라 각 도구가 맡은 범위의 지도**입니다. 빈 칸이나 "by design" 표시는 대개 의도된 설계이고, 반대 방향의 행도 여럿 있습니다(hookify는 더 많은 훅 이벤트를 다루고, Writ는 의미 기반으로 검색하고, Ruler는 30+ 에이전트에 배포합니다 — 셋 다 nunchi에는 없는 능력입니다).

| 능력 | nunchi | hookify | Claude Code 네이티브 룰 | Claude Code 스킬 | Ruler / rulesync | Writ |
|---|---|---|---|---|---|---|
| 행동(명령) 트리거 — `gh pr create`, `git commit` | ✅ | ✅ | 파일 읽기 경로 글롭 | 모델이 풀² | 범위 밖¹ | 부분적 (내장 게이트) |
| 내용 트리거 — 지금 *쓰려는* 것 | ✅ `trigger.field` | ✅ | — | — | 범위 밖¹ | 부분적 (쓰기 전 정적 분석) |
| 차단 시 규칙 본문을 **모델에** 전달 | ✅ 사유로 | Stop ✅ / PreToolUse: 사용자용 메시지 | — | 권고만 가능 — 차단 불가 | 범위 밖¹ | ✅ 룰 ID + 사유 |
| 비차단 JIT 주입 | ✅ `inject` | — | 파일 읽기 시 | ✅ 호출 시² | 범위 밖¹ | ✅ 매 턴 (검색) |
| 전달 상태 — 세션당 1회, 차단 루프 없음 | ✅ | 설계상 무상태 | — | 동일한 재호출은 중복 제거 | 범위 밖¹ | ✅ 단계별 ID + 토큰 예산 |
| 컴팩션 후 재활성화 | ✅ 실측 5/5 | 설계상 무상태 | 루트 CLAUDE.md는 문서화됨; 룰 파일은 불명 | 예산제 재부착 (오래된 것부터 탈락) | 범위 밖¹ | ✅ PostCompact 재주입 |
| source 문서를 전달 시점에 읽음 (어긋남 없음) | ✅ | 메시지가 룰 파일 안에 있음 | ✅ | ✅ 호출 시 본문 읽음 | 생성 시점에 복사 | ✅ |
| 전달 로그 + 리포트 | ✅ JSONL, `/nunchi:report` | — | — | — | 범위 밖¹ | ✅ 마찰 로그 + 대시보드 |
| PreToolUse 밖의 훅 이벤트 (PostToolUse, Stop, UserPromptSubmit) | 아직 — Stop 룰은 로드맵에 | ✅ 4개 이벤트 | — | 스킬 스코프 훅 (frontmatter) | — | ✅ 넓은 커버리지 |
| 의미 기반 룰 매칭 (정규식 아닌 의미) | 설계상 정규식 (결정론) | 정규식 | 경로 글롭 | ✅ 설명 매칭 | — | ✅ 하이브리드 RAG |
| 멀티 에이전트 룰 배포 (Cursor, Cline, …) | 계획 없음 | — | — | ✅ 이식 가능한 SKILL.md (오픈 표준) | ✅ 30+ 에이전트 | — |
| 런타임 의존성 | Python 표준 라이브러리 | Python 표준 라이브러리 | 내장 | 내장 | Node | Docker + Neo4j + FastAPI |
| 공개된 준수 실측 | 위의 사전 등록 실행들 | — | — | — | — | 정성적 압력 실행 트랜스크립트 |

¹ Ruler와 rulesync는 설계상 정적 배포 도구입니다. 설정 파일을 만들어 각 에이전트의 런타임에 넘기는 방식이라, 런타임 전달에 해당하는 행 자체가 적용되지 않습니다.

² 스킬은 모델이 호출합니다. 모델이 요청을 각 스킬의 설명과 대조해서 여는 방식이라, 스킬을 도구 호출에 묶는 장치도, 행동 전에 발동을 강제하는 장치도 없습니다. 호출은 모델의 판단에 달려 있고, [공식 문서](https://code.claude.com/docs/en/skills)는 보장이 필요하면 훅을 쓰라고 안내합니다. 위 FAQ 항목을 참조하세요.

훅 오버헤드 실측: 도구 호출당 중앙값 ~24ms(규칙이 전달될 때 26ms). 각 행은 2026년 7월 기준으로 각 도구의 소스나 공식 문서로 확인했습니다 — 잘못된 곳은 이슈로 알려 주시면 고치겠습니다.

다른 도구들이 잘하는 것도 분명합니다. hookify는 Claude Code 훅 위의 마크다운 frontmatter 룰을 개척했고 더 많은 훅 이벤트(PostToolUse, Stop, UserPromptSubmit)를 다룹니다. 스킬은 절차를 두기에 맞는 자리입니다 — 스킬당 ~100 토큰의 점진적 로드에, nunchi에는 없는 실행 제어(`allowed-tools`, 모델/노력 오버라이드, 분리된 컨텍스트)까지 갖췄습니다. Ruler와 rulesync는 에이전트 간 배포를 제대로 풉니다. Writ는 의미 검색 기반 전달을 탐구합니다. nunchi의 자리는 일부러 좁게 잡았습니다. 결정론적으로, 행동에 트리거되어, 전달 상태를 추적하며 전달하는 것 — 그리고 실측한 것만 주장하는 것.

## 기여

이슈와 PR을 환영합니다. 커밋 메시지는 [Conventional Commits](https://www.conventionalcommits.org/)(`feat:`, `fix:`, `docs:`, …)를 따라야 합니다. [release-please](https://github.com/googleapis/release-please)가 커밋 제목으로 버전과 체인지로그를 만들기 때문입니다.

이 저장소는 nunchi를 스스로 씁니다(도그푸딩). 이 레포의 이슈·PR 규칙이 `.claude/rules/`에 있고, `gh issue create`나 `gh pr create`를 실행하는 순간 훅이 그 규칙을 전달합니다 — 먼저 외워 둘 필요가 없습니다.

README 번역 기여도 환영합니다. 정본은 영어 [README.md](README.md)이고, 번역본은 릴리스 시점에 동기화하는 스냅숏입니다(문서 상단에 기준 커밋을 적습니다).

## 한계와 로드맵

다음은 아직 없는 기능이고, 로드맵에 있습니다.

- **의미 기반 판정**: 정규식 트리거 대신 LLM으로 출력 내용을 검사해 규칙 위반을 잡는 것. 지연과 비용의 트레이드오프 검토가 필요합니다.
- **Stop 이벤트 룰**: 세션이 끝나는 시점에 "과제를 마치기 전에 이 조건이 지켜졌는가"를 검사하는 룰.
- **준수 리포트 UI**: 지금의 `/nunchi:report`는 로그를 표로 집계하는 수준이고, 더 정교한 분석은 백로그에 있습니다.

덧붙여, 위 결과는 조건당 n=3 파일럿과 사전 등록된 압력 예비 실행(6회)에서 나온 것입니다. 통계적으로 "100%"는 "이 표본에서는 실패를 보지 못했다" 이상을 뜻하지 않으며, JIT의 준수 우위 주장은 베이스라인이 실제로 무너지는 압력 수준이 확인될 때까지 하지 않습니다.
